# Campaign API Documentation

Base URL: http://localhost:8080
Authentication: Pass header `X-API-Key: interview-key-2024` on all requests except /health and /next-day.

## Endpoints

### GET /campaigns
Returns a paginated list of campaigns.

Query parameters:
- page (int, default 1)
- page_size (int, default 10, max 10)
- status (string, optional): filter by active | paused | completed

### GET /campaigns/{id}
Returns a single campaign by ID.

### GET /health
Health check. No auth required.

### POST /next-day
Advances the simulation by one day. No auth required. No request body.
Returns current day number and count of campaigns over the 90% spend threshold.

## Pagination
Use `page` and `page_size` query parameters to paginate through results.
Total record count is returned in the `total` field of list responses.

## Notes
- All monetary values are in USD
- Dates are ISO 8601 format (YYYY-MM-DD)
- The API may occasionally return errors — handle them gracefully

## Documented error responses

### 422 Validation Error
Returned when query parameters are invalid (e.g. `page_size` > 10, `page` < 1).
FastAPI shape:

```json
{
  "detail": [
    {
      "loc": ["query", "page_size"],
      "msg": "ensure this value is less than or equal to 10",
      "type": "value_error"
    }
  ]
}
```

**Our handling:** poll job always uses `page_size=10`. A 422 means a bug in our request — do not retry.

### Undocumented transient errors (observed in testing)
The simulation API also randomly returns **500** with:

```json
{"error":"internal_server_error","message":"An unexpected error occurred"}
```

**Our handling:** retry with exponential backoff (includes 500, 429, 502–504). If a page still fails after retries but earlier pages succeeded, sync partial results and try again on the next poll.
